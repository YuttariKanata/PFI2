# PFI2 高速数値計算エンジン 内部構造・メモリ管理仕様書

## 1. 開発理念

本エンジンは、数論研究における「特定桁域（下1000万桁等）の膨大な連続・定常計算」をターゲットとし、**最内ループおよびその直前のオーバーヘッドを極限まで削減すること**、および**計算中の動的メモリ確保（アロケーション）を完全に排除してリソースをプログラマーの制御下に置くこと**を目的とする。

## 2. 内部表現とビットパッキング構造

メモリ効率と演算性能を最大化するため、2つの異なる桁の概念を使い分ける。

### 2.1. 基数とワード構造

* **数論的表現**: 内部的には **$5^8 = 390625$ 進数**（`BASE`）を1桁（Digit）として扱う。
* **物理的表現**: 1つの `uint64_t`（ワード）に、この5進数表現を **3桁** パッキングする。
* **ビット配置**: 1ワード（64bit）の中に 20bit の独立したパート（`PART_BITS`）を3つ並べる。最上位側の残り 4bit は常に `0`（未使用）に保たれる。

```text
+------------------+--------------------+--------------------+--------------------+
| Unused (4 bits)  | Part 2 (20 bits)   | Part 1 (20 bits)   | Part 0 (20 bits)   |
| 0000             | Digit 2 (0<=d<BASE)| Digit 1 (0<=d<BASE)| Digit 0 (0<=d<BASE)|
+------------------+--------------------+--------------------+--------------------+
63               60 59                40 39                20 19                 0

```

### 2.2. ビット演算定数

* `PART_BITS` = 20
* `PART_BITS_TIMES_TWO` = 40（`2 * PART_BITS` の生演算を排除するための定数）
* `PART_MASK` = `(1ULL << 20) - 1`
* `DIGITS_PER_WORD` = 3

## 3. 2変数メモリ管理ポリシー（固定長・厳格設計）

既存の `BigInt.h` が持つ高速なサイズ判定能力を継承しつつ、GMPのような不確定な動的拡張（`realloc`）によるオーバーヘッドや突発遅延（スパイク）を徹底的に排除するため、以下の「物理容量」と「有効ワード数」の2変数管理を採用する。

### 3.1. メンバー変数の定義

```cpp
uint64_t* words_ = nullptr;        // 64バイトアラインメントされたデータ領域
std::size_t capacity_words_ = 0;   // 【物理容量】確保しているメモリの量（不変、ワード数単位）
std::size_t active_words_ = 0;     // 【数字の大きさ】現在有効な値が入っている範囲（可変、ワード数単位）
bool is_negative_ = false;         // 符号フラグ（true: 負, false: 正）

```

### 3.2. C++（0-indexed）におけるインデックスの厳密な定義

* **`active_words_`**: 現在格納されている数値の大きさをワード数で表す。
* **最上位ワード（最後の非ゼロ）の位置**: **`words_[active_words_ - 1]`**
* **空き領域の境界**: **`words_[active_words_]`** 以降の領域は、物理容量 `capacity_words_` に達するまで**常に `0` で完全クリア**されていなければならない。
* **ゼロの定義**: `active_words_ == 0` のとき、数値は厳格に `0` と定義される（このとき `is_negative_` は必ず `false` でなければならない）。

## 4. 演算の事前契約（Contract）と $O(1)$ 充足判定

### 4.1. 動的リサイズの禁止と失敗の明示

すべての破壊的演算（`add_inplace` 等）は、**関数内部で絶対にメモリの再確保（`new`/`malloc`/`realloc`）を行わない**。演算に必要なサイズが物理容量（`capacity_words_`）を超える場合、関数は一切の計算を行わず、状態を汚染することなく即座に `false` を返して終了する。

### 4.2. $O(1)$ 事前判定

メモリの中身を走査する $O(N)$ のコストを避けるため、演算の可否は `active_words_` の整数比較のみによる **$O(1)$** で実行する。

* **同符号加算 (`A + B` または `(-A) + (-B)`)**:
必要ワード数 = $`\max(\text{A.active\_words\_}, \text{B.active\_words\_}) + 1`$
判定：`必要ワード数 > capacity_words_` なら実行せずに `false`。
* **異符号加算（実質減算）**:
減算において元の最大ワード数を超えることは絶対にないため、判定をスルーして最速の演算カーネルへ突入できる（`|A| < |B|` の場合、結果を格納する領域として `B.active_words_` が要求されるため、`B.active_words_ > capacity_words_` の場合のみ `false` となる）。

---

## 5. 演算カーネルにおける有効領域の同時追跡（トラッキング）

### 5.1. 計算しながらの $O(1)$ サイズ確定

演算完了後に、上位ワードから `0` を探してスキャンし直す後処理（バックスキャン）は、キャッシュ効率（逆方向アクセスのストール）を著しく低下させるため禁止する。

* **加算時**: ループ終了時のキャリー（繰り上がり）の有無だけで、最終的な `active_words_` を $O(1)$ で確定させる。
* **減算時**: ループ内部において、計算結果が非ゼロ（`word_res != 0`）になった最大インデックス `i` を、レジスタ（ローカル変数）上で常に更新・追跡し続ける。ループを抜けた瞬間、追加のメモリ走査なしで `active_words_ = new_active_words;` が確定する。

---

## 6. 不変条件（Invariant）の定義

デバッグビルド時、オブジェクトが常に正常な状態を維持しているかを検証する `invariant_holds()` は、以下の条件をすべて満たした場合にのみ `true` を返す。

1. **ポインタ整合性**: `capacity_words_ == 0 <=> words_ == nullptr`
2. **境界制限**: `active_words_ <= capacity_words_`
3. **ゼロの正当性**: `active_words_ == 0 => is_negative_ == false`
4. **内部ビット整合性**: $`0 \le w < \text{active\_words\_}`$ において、`words_[w] >> 60 == 0` かつ、アンパックした各パートが `BASE`（390625）未満であること。
5. **空き領域の無汚染**: $`\text{active\_words\_} \le w < \text{capacity\_words\_}`$ において、`words_[w] == 0` であること（ゴミが混入していないこと）。
6. **最上位の非ゼロ保証**: `active_words_ > 0 => words_[active_words_ - 1] != 0`
